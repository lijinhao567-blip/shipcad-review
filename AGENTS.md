# AGENTS.md

本文件是本项目的 Agent 持久化工作准则。所有参与本仓库的 Agent 在开始任务前，应先阅读本文件，并在实现、调研、文档、测试和架构决策中遵循这里的约束。

## 项目定位

本项目目标是建设一个面向船舶设计院审图组的开源智能审图平台：船舶 CAD 图纸 AI 智能审查与版本管理系统。

项目不应被定位为一次性课程作业原型，而应按可交付、可维护、可验证、可持续演进的开源产品推进。课程资料、报告模板和评分表是背景参考与阶段性交付材料，不能成为降低工程质量、牺牲架构边界或绕开核心技术问题的理由。

当前产品路线：

- Web 审图平台，支持项目、图纸、版本、审查任务、问题闭环、报告和统计看板。
- DXF 优先，CAD Worker 基于 `ezdxf` 进行结构化解析。
- DWG 通过 LibreDWG `dwg2dxf` 转换后复用 DXF 解析链路。
- 正式 DXF 预览使用 `dxf-viewer`，Canvas 仅作为人工诊断视图，不能作为自动兜底掩盖正式预览失败。
- 规则审查优先，AI 负责解释、摘要和报告辅助，不能替代可审计的业务规则。
- 后续通过 YOLOv8、OCR、知识图谱等能力补充证据来源，并统一进入证据层。
- 项目采用 AGPL-3.0 开源路线，默认按可自托管、可私有化部署方向演进。

## 核心工作准则

1. **质量优先**
   不得为了快速实现功能而降低技术标准、绕开核心问题或改变既定产品方向。若当前条件不足以高质量完成，应明确说明限制、风险和下一步补齐方案。

2. **证据驱动**
   涉及关键技术选型、架构调整、外部依赖、AI 方案、CAD 解析、规则引擎、数据集、部署和安全策略时，应优先参考官方文档、成熟开源项目、论文、行业实践或最新可用资料，避免仅凭猜测实现。

3. **方向一致**
   所有实现应服务于项目既定定位：开源、可私有化部署、CAD 结构化解析、规则审查、AI 辅助解释、未来接入 YOLO/OCR/知识图谱。不得为了局部功能便利而破坏总体架构。

4. **架构先行**
   新功能应放在合理模块边界内，例如前端、后端、CAD Worker、Vision Worker、规则引擎、AI Gateway、数据集、部署脚本等。避免把临时代码堆进核心流程。

5. **可验证交付**
   重要功能应配套测试、样例数据、验收脚本或文档说明。不能只做到“看起来能跑”，应能证明它在预期场景下可靠工作。

6. **不做伪智能**
   AI 能力必须建立在明确证据之上。规则、CAD 解析、OCR、YOLO、知识图谱等证据来源应可追踪、可解释。不得用固定文件名、硬编码结果或不可解释输出冒充智能审查。

7. **清晰记录**
   重要决策、限制、实验结果、已知问题和后续计划应写入项目文档。临时方案必须标记为临时，不得伪装成最终架构。

8. **可维护性**
   优先采用清晰、稳定、可扩展的实现方式。避免过度设计，也避免为了省事写出后期难以替换、难以测试、难以理解的代码。

9. **安全与隐私**
   文件上传、鉴权、权限控制、日志、配置、数据存储和模型调用必须考虑安全边界。敏感配置不得硬编码，上传文件必须校验类型、大小和访问权限。

10. **主动说明不确定性**
    如果需求表达不清、技术风险较高、资料不足或实现路径存在分歧，Agent 应主动指出问题，并给出建议路线，而不是擅自假设后继续推进。

## 外部工具与开源依赖准则

- 可以使用开源工具、公开资料、论文、SDK、模型或数据集辅助开发。
- 引入外部工具或依赖前，应说明用途、来源、许可证、安装或下载位置、潜在风险和替代方案。
- 下载或链接外部工具时，应保持路径清晰，优先放入约定目录，避免散落到项目根目录或系统目录。
- 不得引入许可证冲突、来源不明、不可维护或难以复现的依赖。
- GPL/AGPL、商业 SDK、模型权重、数据集许可证等必须明确记录，不得模糊处理。
- 真实船舶图纸、客户资料、私有训练数据、敏感文件和不可公开模型权重不得进入仓库。
- 实验性工具应隔离在 `experiments/` 或明确的适配层中，不能污染正式链路。

## 架构与实现准则

- 优先保持“前端 / 后端 / CAD Worker / Vision Worker / 数据集 / 部署”边界清晰。
- CAD 解析、视觉识别、OCR、知识图谱和 AI 模型等外部能力应通过适配层或独立服务接入，避免核心业务直接绑定单一实现。
- 上传图纸和审查任务必须保持分离：上传负责版本存档，审查负责异步计算和问题生成。
- 项目、图纸、版本、任务、问题、证据和报告接口必须执行项目数据范围校验；全局角色决定可执行操作，`project_member` 决定非管理员可访问的项目。
- 主链路失败应明确暴露错误，不能用兜底方案掩盖问题。例如 `dxf-viewer` 失败时，应优先修复正式预览链路，而不是自动切换 Canvas。
- 新能力应先形成最小闭环，再逐步增强精度、性能和体验。
- 涉及性能目标时，应说明当前目标、测试方式和差距，例如 DXF 文件大小、解析耗时、审查任务耗时和并发能力。

## 数据集与验证准则

- `datasets/rules/expected.json` 是当前规则 golden dataset 的机器可读验收标准。
- `tools/run_golden_e2e.py` 用于验证“登录 -> 建项目 -> 建图纸 -> 上传 DXF -> 发起审查 -> 校验问题 -> 验证文件接口”的端到端链路。
- 系统不得根据文件名返回规则结果；文件名只用于测试定位输入文件。
- 每条规则应至少有一个正常样例和一个异常样例，逐步扩充为可回归的数据集。
- 新增或修改规则、解析器、预览链路、问题生成逻辑时，应优先运行相关测试和 golden E2E。

常用验证命令：

```powershell
.\.venv\Scripts\python.exe -m pytest cad_worker\tests -q
$env:JAVA_HOME=(Resolve-Path .tools\jdk-17).Path
.\.tools\maven\bin\mvn.cmd -f backend-spring\pom.xml test -q
cd frontend-vue
npm run build
```

后端和 CAD Worker 启动后，可运行：

```powershell
.\.venv\Scripts\python.exe tools\run_golden_e2e.py --keep-going
.\deploy\run-access-control-e2e.ps1
.\deploy\run-redis-queue-e2e.ps1
.\deploy\run-object-storage-e2e.ps1
```

## 文档维护准则

- 架构变更应同步更新 `docs/architecture.md`。
- 数据集或验收策略变更应同步更新 `docs/dataset_spec.md` 和 `docs/quality.md`。
- 外部依赖或许可证变化应同步更新 `THIRD_PARTY_LICENSES.md`。
- 开源工具调研和集成策略应优先记录到 `docs/open_source_tools_survey.md`、`docs/integration_strategy.md` 或相关实验文档。
- 用户可见流程变化应同步更新 `README.md` 或 `docs/user_guide.md`。

## Agent 行为要求

- 开始实现前先理解现有代码、文档和数据流，不要凭空重写。
- 尽量使用项目已有模式、命名、目录和技术栈。
- 变更应保持小步、可验证、可回滚。
- 遇到用户方向性要求时，优先保护长期质量和架构一致性。
- 完成后说明改动范围、验证结果、剩余风险和建议下一步。
- 不要把课程报告材料和产品工程材料混在一起；报告可以引用产品架构，但产品不能为了报告短期呈现而牺牲工程质量。
